## Problem Statement

trip-planner 轉 SaaS 方向後，缺**陌生人自主建立行程 + 主動加景點**的 UX 層。目前只有：
- AI 聊天編輯（`/manage` 用 `/tp-edit` skill，門檻高）
- 手動 D1 INSERT（admin-only）

需要一套**使用者自己動手加景點到行程**的 UI pattern。Mindtrip（YC W24）已經解決這個 UX，作為 benchmark 研究他們怎麼做、哪些可 borrow、哪些 trip-planner 可差異化。

## Mindtrip UX Patterns — 11 screenshot 逐點還原

### 1. Information Architecture

**桌機 sidebar（7 個 top-level nav）**：
```
mindtrip. [collapse]
─ Chats (badge count)
─ Trips
─ Explore
─ Saved
─ Updates
─ Inspiration
─ Create
[New chat]  ← primary CTA
─ Traveler [avatar] ···
```

**手機 bottom tab bar（6 tab）**：
```
Chat | Trips | Explore | Saved | Inspiration | Log in
```

**對應 trip-planner**：
- `/chat` ≈ `/manage` AI 聊天
- `/trips` = trip list（trip-planner 現狀無，要建）
- `/explore` = 探索 POI with map（trip-planner 有 `/trip/:id/map` 單行程，缺全局）
- `/saved` = 收藏 POI cross-trip（缺）
- `/inspiration` = 他人 trip 靈感（缺）

### 2. Trip 建立 Flow（3-step "Where to?" modal）

**Image 2, 8, 9, 10**：`+ New trip` or `Create a trip` button → modal 問 3 件事：

1. **Destination**（location autocomplete + dynamic photo → `+ Add location` multi-city + `Road trip?` toggle）— Image 8 選 Oklahoma 後左半 placeholder flamingo 變夜景
2. **Timing**（Dates 兩月 calendar / Flexible 天數+月份卡片）— Image 9, 10
3. **Trip preferences**（自由 text 2000 chars + mic icon voice input）

**Key**：只 Destination 必填，其他 optional。Create 按鈕 gated on Destination。

**Trip 建出來 auto-name**：「Trip to {Destination}」（Image 11：Trip to Okinawa Prefecture）

### 3. Anonymous-friendly gate pattern（關鍵）

**Image 11 右上橙色 banner**：「Sign up to save trip」

**策略**：
- Anonymous 使用者**可建 trip、可加 ideas、可用 chat**
- 只有**跨 session 保存 + 分享**才要 signup
- 登入是 **delayed gate**（progressive engagement），不是 paywall / signup wall

**對 trip-planner 啟發**：
- 目前 trip-planner 是 signup-first（Cloudflare Access 白名單擋）
- Mindtrip pattern = 先讓人用，用到有價值才 signup
- 這是 SaaS conversion 的 standard — Mindtrip 的設計比我們的 CF Access 高明 N 級

### 4. Add-to-trip 核心 UX —— Heart + Plus 雙 action（Image 5）

手機 Explore page 的 POI 卡片 **每張圖右上兩個 icon button**：

```
┌─────────────┐
│ ♥         ⊕ │  ← heart=save / plus=add to trip
│             │
│  [photo]    │
│             │
└─────────────┘
 POI title
```

**Heart**（左上，outline style）= Save to collection
**Plus**（右上，outline style）= Add to trip plan

**兩個 action 分開是 intentional**：
- Heart = 跨 trip 的 global 收藏（類 Google Maps saved / Pinterest）
- Plus = 加到**當前 active trip** 的 Ideas 清單

**對 trip-planner 極有啟發** — 你目前沒有任何 icon button 在 POI 上做這件事。只能靠 AI chat 說「加進行程」。

### 5. Ideas vs Itinerary — 兩個階段分開

**Image 11 右 trip sheet 的 tabs**：
```
Itinerary | Ideas | Bookings | Calendar | Chats | Media
```

**Ideas = "maybe" list**（未決定要不要去、沒綁時間）
**Itinerary = 排好時間軸**（確定要去、綁日期 + 時段）

**UX 流程**：
1. Explore 看到某 POI → tap plus → 進 current trip 的 **Ideas tab**
2. Ideas tab 累積 candidate
3. User 從 Ideas 拖到 Itinerary 排時間（或 AI 自動排）

**為什麼分**：降低 commitment。加到 Ideas 不強迫排時間，沒有「我還不知道哪天去」的摩擦。

**trip-planner 現狀**：只有 Itinerary（`trip_days` + `trip_entries` 綁 start_time），沒 Ideas 概念。所有加入都強制綁日期 → 探索期摩擦。

### 6. Trip sheet split pane（Image 7, 11）

**桌機**：左 sidebar + 中 chat + 右 trip sheet（右 sheet 可 open/close，URL param `tripSheet=1`）

**Sheet 內部 2 層 tab**：
- Top tier（trip metadata）：Destination / When / Traveler / Budget / Preferences
- Bottom tier（content）：Itinerary / Ideas / Bookings / Calendar / Chats / Media

**點 tab → URL 改 `?tripTab=ideas`** — URL-driven state，可直接 share link 到特定 tab。

**trip-planner 對應**：你目前左 timeline + 右 map rail。**右 panel 沒有 tab 切換不同 view**。可借用 right-panel-with-tabs pattern。

### 7. 手機 responsive

**Image 3**（手機 /trips empty state）：
- 無 sidebar（sidebar 變 bottom nav）
- 空狀態同桌機但 layout 較 narrow
- 底部 tab bar 永遠 visible

**Image 4**（手機 "Where to?" modal）：
- **Full-screen cover modal**（不是 bottom sheet）— close X 左上
- Modal 無插畫（節省空間）
- 內容堆疊同桌機但垂直 stack

**手機 Explore（Image 5）**：
- 地圖佔上半 + drag handle + 下半 POI cards horizontal scroll
- **地圖 + cards 共存，不是 tab 切換**
- Search + filter + hide-map 3 icons top

### 8. Auth flow（Image 6）

**登入 modal**（不是 full page）：
- 👋 emoji + "Welcome to Mindtrip"
- Email input → Continue（passwordless？email OTP？）
- 「or」分隔
- Google SSO button（繁中「透過 Google 帳戶繼續操作」— auto-localized）
- Apple SSO button

**對 trip-planner V2 啟發**：
- 你計畫自建 email/password + forgot password → **Mindtrip 只用 passwordless email + 2 個 OAuth**
- 沒有 password input field，沒有 "remember me"，沒有 forgot password link
- **Codex 的 CEO review 當初說的「Google-only 覆蓋 95% demand」在這裡被 Mindtrip 驗證** — 一個 YC W24 AI 旅遊 app **沒做 password auth**

### 9. AI chat ≠ 命令介面（Image 7）

**trip-planner `/manage`** = AI 編輯命令行（「把 Day 3 餐廳換成 XX」）
**Mindtrip `/chat`** = AI 推薦引擎 + 問答

差異：
- Mindtrip 的 "Ask me anything travel related" 是 **discovery**，使用者主動問「沖繩 3 天怎麼玩」
- trip-planner 的 AI 聊天是 **execution**，使用者已經知道要做什麼下指令

**兩個都有價值但解決的問題不同**。Mindtrip 的 chat 是 funnel 入口（免登入就能聊），trip-planner 的 AI 是 power user 編輯工具。

### 10. Delete toast（Image 1 右下）

刪除 trip 後：綠勾 toast「沖繩縣之旅 deleted.」浮現右下，不 modal 打斷。**multi-lingual trip title 被原樣保留**（繁中 trip name 都不 force translate）。

細節：這個 toast 表示 Mindtrip **能刪 trip + 支援 CJK 命名**，trip-planner 你目前繁中 trip name 用 UTF-8 也行（okinawa-trip-2026-Ray 這種 slug ID）。

### 11. Dark mode pure

全部 11 張都是 **dark mode**。沒 light mode screenshot。不確定 Mindtrip 有無 light toggle，但 default 是 dark。

**對 trip-planner V2**：你 DESIGN.md 有完整 Terracotta dark（warm charcoal `#1C140F`），這跟 Mindtrip 的 pure neutral dark（`#000` 附近）風格不同 — Mindtrip 偏 tech-utility，trip-planner 偏 travel-magazine warmth。這是**差異化定位**。

## Trip-planner Gap Analysis（你現在 vs Mindtrip）

| 維度 | Mindtrip | trip-planner 現狀 | Gap |
|------|----------|-------------------|-----|
| Trip 建立 | "Where to?" modal 3-step onboarding | D1 手動 INSERT admin-only | **Critical — 陌生人無法建 trip** |
| Add-to-trip 按鈕 | Heart (save) + Plus (add) 雙 action on cards | **無** — 只能 AI chat 命令 | **Critical — UX entry missing** |
| Ideas vs Itinerary | 兩 tab 分開 | 只 Itinerary 綁時間 | **High — 探索期摩擦** |
| Explore page | 全局 POI browse + filter by category | 無（只單行程 map） | High |
| Saved / Collection | 跨 trip 收藏 | 無 | Medium |
| Anonymous-friendly | 可建 trip 後才 signup | Signup-first (CF Access 牆) | **Critical — funnel 破損** |
| 手機 tab bar | 6 tab (含 Log in) | 4 tab（行程/地圖/訊息/更多） | Low — 已有 |
| Right trip sheet tabs | 6 content tabs | 左 timeline + 右 map rail（無 tabs） | Medium |
| Multi-location trip | `+ Add location` + Road trip toggle | 無 | Low — V2 或後續 |
| URL-driven sheet state | `?tripSheet=1&tripTab=itinerary` | 無 deep link state | Low — nice-to-have |

## 3 個 Adaptation Approach

### Approach A — Minimal：只抄 Heart + Plus action（2-3 天）

最小改動 ship「使用者可以加景點到行程」：

- 在現有 `/trip/:id/map` 或 `StopCard` 加 ⊕ icon button → click 彈出「加到哪個行程？」modal → confirm 後 INSERT `trip_entries`
- 不做 Explore 全局 view，不做 Ideas tab，不做 Saved collection
- 不改 signup flow（繼續 CF Access）
- 只解 "UI entry to add-to-trip" 這一件事

**Pros**：
- 最小 scope，3 天 ship
- 不動 schema（用既有 trip_days / trip_entries）
- 不 block auth plan

**Cons**：
- 只是 AI chat 的 UI alt，不是真的 competitive feature
- Anonymous-friendly 沒解
- 沒 Ideas 探索期 UX

### Approach B — Ideas Layer（2-3 週，Recommended）

借 Ideas vs Itinerary 分層概念，trip-planner 加一個 `trip_ideas` table + Ideas tab：

**Schema**：
```sql
CREATE TABLE trip_ideas (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  trip_id TEXT NOT NULL REFERENCES trips(id),
  poi_id INTEGER REFERENCES pois(id),        -- 可 null（自由文字 idea）
  title TEXT NOT NULL,
  note TEXT,                                 -- 使用者自己的 maybe 備註
  added_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
  added_by TEXT,                             -- email / user_id
  promoted_to_entry_id INTEGER REFERENCES trip_entries(id),  -- 排入 itinerary 時 link
  archived_at TEXT
);
```

**UI**：
- `/trip/:id` 右 panel 加 tabs：`Itinerary | Ideas | Map`（原有 map rail 變 tab）
- Ideas tab = 卡片清單，無時間軸，無 day 綁定
- 每張卡有「排入 Day N」按鈕 → promote 到 `trip_entries` + 設 `promoted_to_entry_id`
- POI card（任何 POI 出現處）加 ⊕ icon → 進 current trip's Ideas

**Pros**：
- 解決**探索期摩擦**（不用選日期就能收集想法）
- 降低「建 trip 就要排完行程」的心理門檻
- 跟 AI chat 互補（AI 建議 → 進 Ideas → 之後排時間）

**Cons**：
- 新 schema + 新 UI + migration
- 2-3 週 scope

### Approach C — Lateral：Pretty URLs + Progressive Auth（1-2 週）

**不抄 Mindtrip UX，抄 Mindtrip 的 anonymous-friendly 理念**，改 trip-planner auth：

- 匿名者可**建新 trip**（localStorage 存 trip_id + 臨時 JWT）
- 匿名 trip 資料進 D1 但 owner_user_id=NULL
- Banner「登入以永久儲存」（抄 Mindtrip Image 11 橙色 banner）
- 登入後 claim：`UPDATE trips SET owner_user_id = ? WHERE id = 匿名 trip_id`
- 取代 CF Access 白名單策略

**Pros**：
- 解 funnel 破損（CF Access 是 0% → 100% 的 hard gate）
- 配合 V2 OAuth server 演化：OAuth-first + anon-friendly
- Mindtrip 證明這 pattern 在 trip planner 可行

**Cons**：
- 改 auth pipeline，跟 V2 OAuth plan 強 coupling
- 匿名資料 GC 策略要想（7 天不 claim 就刪？）

## Recommended Approach

**B + C hybrid**（~3-4 週）：

1. **Phase 1（Week 1-2）— Approach B**：Ideas layer（new schema + tab + ⊕ button）
2. **Phase 2（Week 3-4）— Approach C 的 Progressive Auth**：搭 V2 OAuth 時一起做匿名 trip + claim flow

**為什麼 hybrid**：
- 單 A 太 cosmetic（只是 AI chat 的 UI mirror）
- 單 B 沒解 anonymous funnel（還是登入才能用）
- 單 C 沒解探索期摩擦
- **B + C 一起才是 "Mindtrip 化"** — 陌生人可匿名建 trip、可 collect Ideas、有價值再 signup

**跟 V2 OAuth 的關係**：
- V2 OAuth server = **後端 auth**
- 本 plan = **前端 UX** 配合 V2 auth 的 progressive engagement
- 兩者並行不衝突，但 **C 需要 V2 OAuth 做完才能 claim**

## Key Pattern trip-planner 應該差異化的地方

**不全抄，留 trip-planner 的 unique 訴求**：

| Mindtrip 的 | trip-planner 的 | 為何不同 |
|------------|---------------|---------|
| Pure dark utility（黑） | Terracotta warm（暖焦糖） | Travel-magazine 感 vs tech-utility |
| AI discovery chat（問答） | AI edit chat（命令） | Mindtrip 服務首次規劃者；trip-planner 服務已有計畫的人 |
| 7 sidebar + 6 tab IA | 4 mobile tab IA | trip-planner 簡化優先 |
| Multi-location road trip | 單 trip = 單主題 | 不挑戰 multi-location 的 feature race |
| Inspiration social feed | 無 | 不做 social layer |
| Bookings integration | 無 | 不做交易層 |

**Mindtrip 的方向是「AI-native super app」**，trip-planner 可以不是。你的差異化在**規劃質感 + 旅伴共享**，不必追 discovery 廣度。

## Open Questions

1. **Ideas schema 跟 AI chat 整合**：當 user 在 `/manage` 跟 AI 說「幫我找沖繩必訪景點」，AI 回 5 個 POI → 應該自動進 Ideas tab 還是要 user 手動加？
2. **Ideas 排到 Itinerary 的 UX**：drag-and-drop 還是 dropdown「排到 Day 3 09:00」還是 AI 自動？
3. **手機 `/explore` 要不要做**：現在有 `/trip/:id/map` 但沒全局 Explore。做不做？scope 還要再加 1-2 週
4. **匿名 trip GC 策略**：7 天不 claim 就刪？留永久 orphan？
5. **⊕ 按鈕的 `current trip` 如何決定**：user 有多個 trip，按 ⊕ 要彈選 trip 還是自動加到 "最近開的"？
6. **Ideas ↔ trip_pois 的關係**：Ideas 是 POI reference 還是 POI full snapshot？（影響 POI 被改時 Ideas 要不要跟著改）

## The Assignment（48h 內）

使用者 48h 內做這件事：

**打開你現有的一個 trip（`okinawa-trip-2026-Ray`），用 Mindtrip 的思維重新走一次**：
1. 哪些 POI 目前在你 itinerary 是 "確定去"？標出來
2. 哪些是 "還在考慮" 但已被你排進時間？那些就是應該在 Ideas 而非 Itinerary 的
3. 數一下：如果你有 Ideas tab，會從現 itinerary 挪幾個過去？

這個練習會讓你**感覺到 Ideas / Itinerary 分層的價值是不是真的存在**。若答案是「0 個，我的 trip 都是確定去的」→ Ideas tab 對你個人用途沒價值，Phase 1 可砍 scope。若 ≥ 5 個 → Phase 1 值得做。

## What I noticed about how you think

1. **你跳過 Mindtrip 的 chat SPA 卡死後堅持走 C（debug browse）而非 fallback**：這跟你連續 3 輪 override autoplan consensus 是同一個 pattern — **遇到阻礙傾向 push through 而不是 work-around**。在 debug infra / 競品 screenshot 這種**手段**層面，這 push-through 浪費時間（10+ 分鐘 debug browse 換來 0 screenshot，不如 3 分鐘自己截）。在**目標**層面，這個 push-through 是 founder 性格。區分手段 vs 目標的 push-through 是 productive 判斷。

2. **你主動去研究 Mindtrip**：autoplan CEO review 時兩 model 都點名 Mindtrip 作為 YC W24 competitive threat。你**主動去 benchmark** 他們，這是對 review 的 productive 回應 — 不是 defensive 抗拒，是吸收資訊。這是 founder signal 中 **"domain expertise + taste"** 的結合。

3. **你截 11 張包含「沖繩縣之旅 deleted」toast**：這個 detail 不是主角但你保留。顯示你的 screenshot discipline 是**全景 capture**，不只抓重點 — 這對 UX research 特別有用，因為 peripheral UI 常常藏著 pattern（如 delete 確認用 toast 不 modal）。良好的 research instinct。

4. **你沒問「那我該做哪個 Approach」**：你讓 doc stand，沒立刻決策。這跟 V2 autoplan 時你很快 override 不同 — 看來 **UX design decisions 你比 arch decisions 更願意 deliberate**。這是合理的，UX 決策的 reversibility 比 schema migration 低，多想一天 cheap。
