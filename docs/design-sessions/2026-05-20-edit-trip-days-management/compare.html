EditTripPage「天數管理」mockup — 3 variants 比較

需求：增加/減少行程天數；移除有景點的天 → 紅色警告 + confirm dialog（確認後 cascade 刪除該天所有 entries）。
共同 SoT：terracotta tokens · DESIGN.md · 既有 ConfirmModal pattern · destructive 走 --color-destructive。

V1 · Days list with per-day delete Mobile fullpage

每天獨立一張 row card，含 date + entry count + ✕ button。「+ 加入一天」append 在 list 結尾。任意天可移除（中間刪 → 後續 day_num 重編）。

IA decisions

新增天
列表結尾「+ 加入一天」append，date 接續上一天 +1
移除天
每 row 自己的 ✕，empty / has-entries 都跳 confirm（has-entries 紅色 border + 紅色 button）
Shift 語意
中間天移除 → 後續天 day_num 重編、日期 cascade 上移
Confirm 內文
「Day N 是 M/D（週X）目前有 K 個景點」+ shift 提醒 warning box
Dirty state
新增/移除 → 「儲存變更」啟用，新天 row 套 .is-new highlight
優點
  • 每天獨立可見 — 5 天 → 5 row，user 看到「正在改 Day 3」直覺
  • 任意天移除（不只結尾）— 中間插入的 day 可以單獨刪
  • 視覺對齊「目的地」list pattern（同樣 row + handle + remove）
缺點
  • 7 天 trip → 7 rows 占垂直空間，scroll 較長
  • 中間天移除 + 日期 cascade 語意對 user 可能複雜（需 warning 解釋）

V2 · Counter +/- Mobile inline

緊湊 counter `[−] 5 [+]` 風格。只能從尾巴增減。− 時若最後一天有景點，展開 inline summary +「將刪除 Day N · X 個景點」紅色 row → 點確認跳 modal。

IA decisions

新增天
「+」直接 append，date 自動順延（不需 confirm）
移除天
「−」嘗試刪最後一天；empty 直接刪，has-entries 展開 inline summary + 跳 confirm modal
Shift 語意
不存在（只動結尾，前面 days 不變）— UX 最直覺
Confirm 內文
「Day N 是 M/D（週X）目前有 K 個景點」（不需 shift warning）
Dirty state
counter value vs original 不同 → dirty
優點
  • 佔位最小（1 row）— 9 成 user case「想加減幾天」滿足
  • 無 shift 語意 — 一律從尾巴動，user 不需理解 day_num 重編
  • 實作最簡單 — backend 只需 POST/DELETE last day
缺點
  • 不能單獨移除中間天 — 想刪 Day 3 必須先刪 4/5 再加回
  • 每 day 的 metadata（景點數）看不到，user 不清楚刪掉的是空 day 還是滿 day（需 click − 才浮出）

V3 · Desktop two-pane Desktop 1024+

桌機 form 左、days pane 右 sticky。每天 compact row + 全域 [−] [+] 上方 + 任意 row ✕ 移除。Mobile 自動 fallback V1 stacked。

IA decisions

桌機 layout
1024+ 兩 pane：左 form / 右 sticky days pane
新增天
頂部 counter 「+」append；任意 row 沒有「插入」action（避免複雜化）
移除天
每 row ✕（任意天）+ 頂部 counter 「−」（捷徑刪最後一天）兩種入口
Shift 語意
中間刪 → 後續 day_num cascade 上移（同 V1）
Mobile fallback
≤1023px 切回 V1 stacked layout（days 區塊 inline）
優點
  • 桌機長 form 不被 days 列表撐長 — 右 sticky 不滾動
  • 兩種 affordance — 快速 counter +/- 或精準 per-row ✕
  • 跟既有 trip detail 桌機 right map sticky pattern 一致
缺點
  • 兩種入口可能造成混亂（counter「−」vs row ✕ 行為差一個刪最後、一個任意）
  • 桌機 sticky pane 在 narrow viewport 1024 邊緣可能擠（右 380px）

建議選 V1（per-day list with shift），原因：

  1. User 需求「移除有行程的天要紅色提醒」明確指向「per-day」決定，counter 風格的 V2 無法表達「移除 Day 3 而非 Day 7」
  2. 視覺對齊既有「目的地」list pattern — user 學一次 list-with-remove 行為，跨 sections 重用
  3. 5–7 天 trip vertical scroll 可接受（既有 trip detail 也是長 timeline scroll）
  4. 桌機可後續加 V3 two-pane 優化 — 先把 V1 ship，V3 是後續 layout polish 不影響功能

V2 適合「我只想簡單加幾天」case，但本需求重點是「移除有景點的天要警告」— V2 把所有 days 隱藏在 counter 後，user 看不到誰被刪。
