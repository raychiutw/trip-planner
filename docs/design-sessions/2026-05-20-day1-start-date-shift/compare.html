「Day 1 起始日期」整體平移 — 3 variants 比較

需求：「修改天數提供另外一種模式 直接設定 Day 1 的起始日期」
共同 SoT：terracotta tokens · 平移後保留 entries 時段（只變日期）· 後端走 PATCH /trips/:id with `dayShift` 或 `startDate` field。

V1 · Inline 起始日期 row Mobile

「行程天數」section 頂部固定一個 inline date input row，user 改 date → confirm modal → 平移。

IA decisions

位置
section 頂端，「加一天」card 上方
觸發
change event 直接跳 confirm modal（避免誤觸）
Confirm 文字
「行程將從 5/1-5/5 平移至 8/15-8/19」
儲存語意
立即送 backend；entries 不變只變 dates
視覺權重
白底 + 一行 input；不搶 add/remove 焦點
優點
  • 最直接 — 永遠看得到、永遠可改
  • 無需多一個 layer（modal）— 1 tap 即可修改
  • 對齊既有「目的地」、「行程名稱」inline pattern
缺點
  • 多佔 56px 垂直空間（即使 user 不常用）
  • 誤觸 risk（user 想 scroll 但點到 input）— confirm modal 防禦

V2 · Button + Modal Mobile

「移動整個行程」card-style button → 點開 modal 含 date input + 即時 preview「5/1 → 8/15」+ 確認。

IA decisions

位置
section 頂端，calendar icon button card style
觸發
tap button → open modal
Modal 內
date input + 即時 preview「old → new」+ 取消/確認
儲存語意
modal 確認後送 backend
視覺權重
icon + 2 行 label + chev right；視覺像 settings row
優點
  • 清楚標示「這是 destructive-ish action」需要確認
  • Modal 內可放 preview 文字幫 user 確認
  • 不會誤觸（明確 tap → modal）
缺點
  • 多一步 modal interaction（比 V1 慢）
  • 當前日期看不到（要先點開 button 才知道現值）

V3 · Tap Day 1 date inline Mobile

不加新 control — Day 1 row 的 date 本身變 tappable（dotted underline 提示），tap 開 popover 修改。

IA decisions

位置
Day 1 row date 本身（reuse 既有 element）
觸發
tap Day 1 date → 開 popover
Popover 內
date input + 即時 preview + 取消/確認
視覺提示
Day 1 date dotted underline + ✏ icon hint
儲存語意
popover 確認後送 backend
優點
  • 零新空間佔用 — reuse 既有 Day 1 row
  • 直覺 — 想改起始日期就點起始日期
  • 對齊 macOS / iOS inline edit pattern
缺點
  • Discoverability — user 可能不知 date 可點（dotted underline + ✏ 不一定夠）
  • Day 2-5 dates 不可點 — 增加 inconsistency
  • 誤觸 risk — 想 scroll 點到 row → popover 開啟

建議：V2 (Button + Modal)，原因：

  1. 明確標示 destructive-ish action — 平移整個 trip 是大改，user 需要 confirm
  2. Modal 可放即時 preview「5/1 → 8/15」幫 user 看見效果
  3. discoverability vs noise 平衡 — button label「移動整個行程」清楚，不像 V3 要靠 dotted underline 提示
  4. 不誤觸 — 點 button 才能改，V1/V3 都有可能誤觸
  5. 缺點「多一步 modal」實際上是 feature — destructive action 應該慢一點

V1 適合「user 常常改 date」case，但實際只在 trip 開始前 / 重新規劃時用，慢一點沒差。
V3 適合「inline editing 文化重」的 app，但 Tripline 其他地方都是 navigate-to-edit 為主，不一致。
