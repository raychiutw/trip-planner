# Runbook — v2.33.x migration deploy 順序

**Last updated**: 2026-05-24 (v2.33.69)
**Scope**: migrations 0067 → 0071 deployment + 配套 schema-touching code

D1 migrations 與 CF Pages backend deploy **非原子**。Schema change 必須對齊 code
deploy 順序，否則「backend 讀已 DROP 欄位」或「new code 寫不存在的表」會 silent
fail 進 `context.waitUntil()`。

## Deploy 順序 — additive FK migrations (0069 / 0071)

**Pattern**: Schema additive — 加新 col + FK constraint，無 DROP。

1. Apply migration first (no backend change blocks on it):
   ```bash
   wrangler d1 migrations apply trip-planner-db --remote
   ```
2. Verify backfill row count + FK structure:
   ```bash
   wrangler d1 execute trip-planner-db --remote --command \
     "SELECT COUNT(*) FROM audit_log WHERE changed_by_user_id IS NOT NULL"
   # Should match users-with-email row count
   ```
3. Merge backend PR (no backend code reads new col yet — pure forward compat)

**Rollback**: 拔 col 需另一 swap migration (D1 不支援 DROP COLUMN FK)。Worst case
restore from D1 backup.

## Deploy 順序 — sqlite_sequence fix (0070)

**Pattern**: Pure data idempotent — `INSERT OR REPLACE INTO sqlite_sequence`。

1. Safe to apply anytime (重跑 0 effect)
2. Apply via `wrangler d1 migrations apply`
3. Verify per affected table:
   ```bash
   wrangler d1 execute trip-planner-db --remote --command \
     "SELECT name, seq FROM sqlite_sequence WHERE name IN ('trip_days', 'trip_entries', 'trip_destinations', 'trip_docs', 'trip_doc_entries')"
   ```
4. Each `seq` 應 = MAX(id) of corresponding table

## Deploy 順序 — DESTRUCTIVE migrations (DROP COLUMN / DROP TABLE)

**Rule**: 必須 2-phase split:
- **Phase 1**: deploy backend with dual-read fallback (新 path + 舊 path 並存)
- **Phase 2**: wait observation period (例 7 天)
- **Phase 3**: apply DROP migration + remove dual-read

例 v2.29.x migration 0061→0062 `trip_pois` DROP 走 2-phase；migration 0068 DROP
default_travel_mode 走 cutover (確認無讀者後 DROP)。

## CF Pages auto-deploy race window

CF Pages 收 master push 後 60-120s build。`.github/workflows/deploy.yml` migrate
job 跑 ~10-30s — common case migrate 先 land。Not guaranteed serial:

- If migrate fails: Telegram alert (deploy.yml line 80-91) — 立刻 manual
  `wrangler d1 migrations apply` retry 或 revert PR
- If CF Pages build fails after migrate succeed: backend rollback 但 schema 已新 →
  舊 code 讀新 schema 通常 backward-safe (additive)
- If migrate succeed + CF Pages succeed but new code has bug: rollback PR via
  revert commit → schema 仍新但 backend revert，注意 destructive migration 不可
  revert
