# 專責角色質疑視角

舊版 Challenger 的 11 視角已分散到各專責角色。每個角色在自己的審查中，除了執行檢查項目外，**必須主動質疑**該視角的問題。

## 視角分配

| # | 視角 | 挑戰什麼 | 負責角色 | 範例問題 |
|---|------|---------|---------|---------|
| 1 | 📋 需求 | 方向對不對 | **Architect** | 使用者真的需要這功能嗎？ |
| 2 | 💻 程式 | 怎麼寫的 | **Reviewer** | 為什麼不用 React.lazy？ |
| 3 | ✅ 品質 | 測得夠不夠 | **QA** | 網路斷線時有測嗎？ |
| 4 | 🔒 資安 | 安全不安全 | **Security** | config.yaml 的 token 該移除嗎？ |
| 5 | ⚡ 效能 | 快不快 | **Architect** | 首頁 LCP 為什麼 2 秒？ |
| 6 | 🐛 漏洞 | 依賴安全嗎 | **Security** | marked.js 有 CVE 嗎？ |
| 7 | ♿ 無障礙 | 所有人能用嗎 | **QA** | speed dial 有 aria-label 嗎？ |
| 8 | 🌐 相容性 | 哪都能跑嗎 | **QA** | Safari 的 dvh 有支援嗎？ |
| 9 | 📊 資料完整 | 資料對不對 | **QA** | 匯出 MD 有包含購物推薦嗎？ |
| 10 | 💰 成本 | 花不花錢 | **Security** | D1 每日寫入量會超 100K 嗎？ |
| 11 | 🎨 UX 設計 | 動線合理嗎 | **Designer** | 這個操作步驟太多了嗎？ |

## 質疑原則

- 每個角色在審查時，**主動從自己負責的視角提出質疑**
- 質疑寫入各自的報告檔（review-report / qa-report / security-report / architect-report / design-report）
- 嚴重度標記：🔴高 / 🟡中 / 🟢低
- 🔴 高嚴重度 → PM 呈報 Key User 裁決
- 🟡 中嚴重度 → PM 通知 Key User
- 🟢 低嚴重度 → PM 自行處理

## 質疑報告格式

```markdown
### {日期} — {角色}: {視角}
- **問題**：{質疑內容}
- **嚴重度**：🔴高 / 🟡中 / 🟢低
- **建議**：{如果有的話}
```

## 為什麼分散質疑比集中質疑好

1. **專業深度**：Security 問安全比 generalist 更到位
2. **並行效率**：不需等一個 Challenger 跑完 11 視角
3. **即時發現**：各角色在自己的審查階段就能發現問題，不用等到最後
4. **責任明確**：每個視角有明確 owner，不會有「以為別人會看」的盲點
